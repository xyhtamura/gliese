# Gliese Delay — Handover

A physics-derived multi-tap delay/reverb. Taps are generated by ray-tracing sound
through a spherically-symmetric graded-index channel, not dialed in by hand. Based
on an undergraduate physics thesis (ray propagation in an RHO refractive-index
profile, Lagrangian/ray-particle analogy). Theme: "acoustic propagation through
exotic planetary atmospheres" — hence **Gliese** (the Gliese catalogue of nearby stars).

## Core idea (the bridge)
A source in this channel reaches a receiver via many ray paths (one per launch
angle). Each path has its own travel time + attenuation → the set of arrivals
`{(delay τᵢ, gain aᵢ)}` **is an impulse response**. Convolve input with it = the
channel's echo/reverb. The thesis ray tracer is therefore an IR generator.

- ray path → delay tap
- launch-angle fan → tap density
- physical knobs (r₀, ω₁/ω₂, boundaries, listener range) → generate all taps coherently

## How it differs from a normal multi-tap
Sample-level DSP is identical (`Σ aᵢ·x(t−τᵢ)`). The difference is upstream:
1. **Generative macro-control** — ~5 physical knobs generate ~hundreds of structured taps.
2. **Characteristic arrival envelope** — deep-channel signature: resolved early taps
   building to clusters/crescendos, not evenly-spaced or simply-decaying. Falls out for free.
3. **Caustics** — many paths focusing at one instant → emergent amplitude spikes.
4. **Coupled modulation** — sweeping one physical knob slides all taps along physical
   trajectories together (converging/diverging, caustics sweeping). One-parameter
   automation of motion a normal delay can't do.

Honest limit: few taps + static settings ≈ indistinguishable from a hand-tuned delay.
Distinctiveness needs density (many rays) + motion (sweeping knobs).

## Design decisions locked in
- **RHO profile** (analytic), not CVG/RK4, for v1. Symmetric channel first (ω₁ = ω₂).
- **Partially reflective boundaries** (coeff ~0.6) so there's a decaying tail (pure
  absorbing = SOFAR crescendo-then-cut, no tail).
- **Finite receiver sphere** (not a point) so finitely many rays register; each pass = a tap.
- **Delay = optical path** `L = ∫n ds` (∝ acoustic travel time), auto-scaled to a
  target first-arrival in ms so musical timing is a knob, not model units.
- Physical realism over usability guardrails: **not every "planet" sounds good, and
  that's accepted.** Musically-useful settings come later as named presets, not as
  constraints baked into the model.

## Physics specifics
- `n²(r)` peaks at r₀. Outer shell (r>r₀): harmonic, force `−C2·r`, ω₁=√C2 → cos/sin motion.
  Inner shell (r<r₀): inverse-harmonic, force `+C3·r`, ω₂=√C3 → sinh/cosh. Rays oscillate about r₀.
- Profile re-parameterised vs thesis (OCR'd continuity constant was garbled): same
  peak-at-r₀ shape, same force signs, same ω's. Trajectory is independent of the n²
  offset (thesis confirms), so only delay/gain weighting is affected.
- Eikonal consistency: |v| = n(r) everywhere; source launches rays at speed n(r_source).

## Current code — `gliese_delay.py` (Python: numpy, scipy, matplotlib; needs ffmpeg)
Single script, runs end to end. Stages:
1. `World` class — RHO profile + `accel(pos)` (the ray-particle force).
2. `trace_bundle()` — velocity-Verlet, vectorised over the whole ray bundle;
   specular reflection at boundaries; accumulates optical path L + reflection count.
3. `render_static()` — 3D ray bundle, coloured by travel time.
4. `render_wavefront()` — animated 3D wavefront (Appendix A in 3D) → mp4.
5. `extract_taps()` — every fresh entry into the receiver sphere = one tap; gain =
   spreading × refl^bounces, polarity flips per bounce.
6. `taps_to_ir()` — bins taps into a sample buffer (caustics = coincident taps summing),
   light Hann band-limiting.
7. `dry_signal()` + convolve → dry/wet WAVs (48 kHz).

Key params (in `World.__init__` and `main`): `r0, a, b, C2, C3, peak, refl`,
receiver angle/radius, `target_first_ms`. Change → whole tap structure regenerates.

Last run: symmetric world, 1800 rays → 296 taps, 70 ms → ~2 s. IR shows clustered
arrivals (successive oscillation passes) — the intended fingerprint.

Outputs: `gliese_wavefront.mp4`, `gliese_ray_bundle.png`,
`gliese_impulse_response.png`, `gliese_wet.wav`, `gliese_dry.wav`.

## Known v1 placeholders (swappable, not load-bearing)
- Geometric spreading model is crude (`1/(1+L/L0)`).
- Per-bounce polarity flip is a toy.
- No diffuse late-reverb stage → tail stays discrete, doesn't smooth into a wash.
- Static-plot colour saturates toward long-bouncing rays.

## Roadmap / next moves
- **Asymmetric "alien" world** (ω₁ ≠ ω₂) — hear non-Earth geometry skew the taps.
- **Custom audio input** — convolve a user's file instead of the synth pluck.
- **Planet presets** — map parameter regions to named worlds:
  - strong ω₁ + thin outer shell → tight flutter / metallic comb
  - ω₁ ≪ ω₂ (asymmetric) → sparse, irregular, lopsided taps
  - ω₁ ≈ ω₂ + thick shells → smooth dense cluster (closest to real reverb)
- Later: better spreading/loss model, diffuse tail (FDN behind the taps), real-time
  port (Web Audio offline processor first, then JUCE/nih-plug VST if wanted in a DAW).

## Thesis reference
"Application of the Lagrangian Formulation in Ray Propagation in a Medium with a
Spherically Symmetric Index of Refraction Profile" (BS Physics, UP Diliman, 2012).
RHO profile in ch.1–2; ray-particle analogy `V(r) = −½n²(r)`; SOFAR-channel motivation;
Appendix A = ray pulse evolution; Appendix B = CVG profile w/ real ocean numbers.
